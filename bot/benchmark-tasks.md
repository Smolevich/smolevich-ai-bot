# Как мы оцениваем free-tier LLM-модели

Цель benchmark'а — публично, воспроизводимо и без человеческих оценок сравнить
бесплатные LLM-модели, которые мы подключаем в [Smolevich AI Bot](https://t.me/smolevich_ai_bot). Лидерборд
показывает результат прогона, методика лежит здесь.

## Что измеряем

Раз в сутки для каждого провайдера (OpenRouter, Groq, Cerebras, NVIDIA,
Hugging Face) отбираются три самые стабильные free-моделей за последние 7 дней
по `model_health_log` (>= 6 проверок, success rate >= 0.9, средняя latency
≤ 8 секунд). Эти три модели проверяются в native-режиме чата. **Топ-1**
модель каждого провайдера дополнительно проверяется в claude-режиме (с
доступом к инструментам Read/Write через `acpx`).

## Задачи

| Задача | Режим | Датасет | Лицензия |
|--------|-------|---------|----------|
| `gsm8k_native` | native | [GSM8K test](https://huggingface.co/datasets/openai/gsm8k) (15 случайных задач в репо) | MIT |
| `gsm8k_tooluse` | claude | те же GSM8K | MIT |

GSM8K — школьная математика, где ответ всегда число. Это насыщенный бенчмарк
для топ-моделей, но для free-tier сегмента (Llama-3 8B/70B, Qwen 2.5,
Gemma 2, Mistral Small) он по-прежнему хорошо различает: 8B-класс выдаёт
50–60% pass rate, 70B-класс 85–93%, слабые — менее 40%.

## Скоринг

### `gsm8k_numeric` (native)

Модели даётся условие задачи и инструкция «в конце выведи строку
`ANSWER: <число>`». Из ответа поочерёдно пробуем извлечь число паттернами:

1. `####\s*(-?[\d.]+)` (формат самого GSM8K),
2. `\boxed{...}` (распространённый LaTeX-маркер),
3. `ANSWER:\s*(-?[\d.]+)` (что мы попросили),
4. Последнее число в тексте.

Извлечённое число сравнивается с эталоном с допуском 1e-3. Если совпало —
`ok=True`, `score=1.0`, иначе `score=0.0`.

### `gsm8k_tooluse` (claude)

Та же задача, но модель работает через `acpx` в подовом sandbox'е и обязана
записать рассуждения в `scratch.md`, итоговое число — в `answer.txt`,
после чего ответить словом `DONE`. Wall-clock ≤ 120 секунд, не более
3 вызовов инструментов (защита от tool-loop'ов).

Скор:

- 0.5 за наличие непустого `scratch.md` (маркер фактического использования инструментов),
- 0.5 за корректное число в `answer.txt` (то же сравнение, что и в native),
- `ok=True` только при полном 1.0.

## Общий рейтинг

Для каждой модели считаются три числа: pass rate в native, pass rate в
claude, средняя latency успешных запросов. Итоговый скор:

```
overall = 0.45 * health_rate + 0.45 * bench_score + latency_bonus
bench_score = 0.65 * native_score + 0.35 * claude_score  (claude_score=0 если не гонялось)
latency_bonus = max(0, min(0.1, (6000 - latency_ms) / 60000))
```

`health_rate` — фактор стабильности из `model_health_log` (success rate за 7
дней). Latency бонус ограничен ±0.1 — это косметика, основной вес у
надёжности и качества решений.

Статус `unstable` выставляется если health_rate < 0.75 или bench_score < 0.6.

## Что НЕ покрывает текущий бенчмарк

- Многоходовые tool-use сценарии (только запись/чтение пары файлов).
- Длинный контекст (промпты ≤ 1k токенов).
- Языковую специализацию (RU/EN — лидерборд не разделяет).

GPQA-Diamond, MMLU-Pro и более тяжёлые задачи добавим позже без миграций — структура
`benchmark-tasks.json` это позволяет.
